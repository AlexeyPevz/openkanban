## Восстановленный product/spec prompt

```text
Продукт
Название: Kanban Plugin for Agentic IDEs

Рабочая идея: файловый канбан-плагин, который встраивается в IDE, сам находит источники задач, отображает карточки в нативном стиле среды и позволяет запускать/контролировать агентский workflow через перемещение карточек между колонками.

Первая целевая среда: OpenCode Desktop, потому что ты уже строишь там свой harness, и сам плагин должен жить внутри этой экосистемы как родной элемент интерфейса.
Вторая цель: позже обобщить архитектуру так, чтобы адаптеры можно было сделать и под другие agentic IDE/CLI-инструменты.

Принципы
Плагин не должен требовать ручной разметки всего проекта перед стартом; он должен сам обнаруживать стандартные источники данных, конфиги и task-файлы по набору правил и fallback-стратегий.

Плагин должен имитировать родную среду, а не выглядеть как отдельное приложение внутри IDE.

UI должен быть предельно минималистичным: чёрно-белая подача, спокойные прогресс-бары, те же шрифты, та же тема, та же плотность интерфейса, что и у хоста.

Действия в канбане должны быть операционными, а не декоративными: перенос карточки меняет состояние в файлах и может запускать оркестрацию задачи.

Базовая версия должна работать как file-based source of truth, чтобы всё было прозрачно, git-friendly и удобно для open-source публикации.

Проблема
Сейчас агентские среды часто дают только режим наблюдения: можно видеть, что что-то происходит, но неудобно вмешиваться в порядок работ, переназначать задачу, параллелить действия или быстро понять, где блокер. В твоём случае это особенно критично, потому что harness ещё достраивается, а тебе уже нужен более управляемый слой поверх агентов и текущих проектов.

Цель MVP
Сделать встроенный kanban, который:

сам находит task-источники в проекте;

рендерит карточки и колонки в стиле хоста;

обновляется от изменений файлов в реальном времени;

при перемещении карточки меняет её статус в source-of-truth;

может передавать событие оркестратору, чтобы задача реально стартовала.

Не-цели MVP
Полноценный Jira/Linear-аналог.

Сложная аналитика, отчёты, burndown, velocity.

Богатая кастомизация UI.

Полноценный SSH/SFTP-слой внутри канбана; это отдельная идея, которую ты уже отложил на потом.

Источники данных
Плагин должен поддерживать режим автообнаружения, где он проверяет несколько типовых источников по приоритету:

.tasks/board.yml

.tasks/tasks.yml

.kanban/board.json

tasks/**/*.md

issues/**/*.md

AGENTS.md, RULES.md, README.md как дополнительные источники контекста

конфиг IDE/проекта для поиска агентов, шрифтов, темы, hotkeys

Если найдено несколько источников, плагин должен:

определить основной source-of-truth,

показать пользователю, что именно выбрано,

позволить переопределить источник позже в настройках.

Модель данных
Минимальная сущность карточки:

id

title

status

description

priority

assignees

required_agents

required_skills

progress

blocked_reason

artifacts

source_file

updated_at

Минимальные статусы:

planned

active

review

done

blocked

Дополнительно:

parked

cancelled

Поведение оркестрации
Ключевое правило MVP: карточка — это не просто визуальный объект, а триггер процесса.

Примеры:

planned -> active = задача готова к запуску;

active -> review = работа завершена и ждёт проверки;

active -> blocked = процесс упёрся в блокер;

blocked -> active = блокер снят, можно продолжать.

Варианты интеграции:

MVP: плагин пишет изменение статуса в файл, а внешний оркестратор сам подхватывает это изменение.

V2: плагин дополнительно шлёт событие в runtime/host API, чтобы оркестратор стартовал сразу после drag-and-drop.

UI
UI должен наследовать визуальную систему хоста:

шрифты — из текущих настроек IDE;

светлая/тёмная тема — автоматически из хоста;

spacing, радиусы, контрасты — максимально близкие к нативным;

чёрно-белая минималистичная подача без лишней цветовой сигнализации.

Карточка в MVP:

заголовок;

маленький статус;

прогресс-бар;

иконки/метки назначенных агентов;

индикатор блокера;

tooltip с деталями при hover.

Tooltip должен показывать:

текущее состояние;

что именно делает оркестратор;

какие агенты задействованы;

если зависло — короткое объяснение причины.

Горячие клавиши
Так как это desktop workflow, плагин должен открываться и управляться с клавиатуры.

MVP hotkeys:

открыть/скрыть канбан;

перейти к следующей/предыдущей карточке;

быстро сменить статус;

открыть детали карточки;

обновить board scan.

Команда вызова должна быть абстрагирована через host adapter, чтобы в OpenCode она бралась из его командной системы, а в других средах могла быть переопределена.

Архитектура
Рекомендуемая архитектура:

Core engine — парсинг board/tasks, нормализация данных, file watching, правила переходов.

Host adapter — интеграция со шрифтами, темой, hotkeys, command palette, runtime events.

Renderer — UI-слой с колонками, карточками, tooltip и sidebar.

Orchestrator bridge — отправка событий о смене статусов.

Discovery service — поиск файлов и определение source-of-truth.

Такой расклад позволит сначала сделать OpenCode-адаптер, а потом не переписывать core при переносе в другую среду.

User stories
Для пользователя
Как пользователь, я хочу открыть kanban внутри IDE и сразу увидеть задачи проекта без ручной настройки.

Как пользователь, я хочу перетаскивать карточки между колонками и быть уверенным, что это меняет реальные файлы проекта.

Как пользователь, я хочу видеть, что делает оркестратор по задаче, не залезая в отдельные логи.

Как пользователь, я хочу быстро понять, где задача застряла и почему.

Как пользователь, я хочу управлять канбаном с клавиатуры, не только мышкой.

Для разработчика harness
Как разработчик harness, я хочу, чтобы kanban работал поверх файлов, а не поверх закрытой базы данных.

Как разработчик harness, я хочу, чтобы plugin автоматически находил board/task-файлы по стандартным путям.

Как разработчик harness, я хочу связать переходы карточек с агентским runtime и оркестрацией.

Acceptance criteria
Плагин стартует без ручной конфигурации, если в проекте найден хотя бы один поддерживаемый task-source.

Плагин корректно наследует тему и шрифты хоста.

Drag-and-drop карточки меняет source-of-truth в файле.

File watcher обновляет UI без перезапуска.

Hover показывает статус, прогресс и блокеры.

Есть hotkey для открытия канбана.

При переводе в active создаётся событие, доступное для оркестратора.

Backlog после MVP
Swimlanes по агентам или проектам.

Параллельные подзадачи.

История изменений карточки.

Inline-редактирование шагов.

SSH/SFTP-backed boards для серверных проектов, если ты вернёшься к этой идее позже.

Поддержка нескольких host adapters помимо OpenCode.

Командная палитра: “Move task to Active”, “Assign agent”, “Explain blocker”.

Формат задач для разработки
Я бы разбил реализацию так:

BRD — зачем продукт нужен и какую боль снимает.

PRD — поведение, данные, UI, acceptance criteria.

User stories — сценарии пользователя и harness-разработчика.

Technical design — discovery engine, parser, adapter API, file watcher, orchestrator bridge.

MVP task list — конкретные issue для кодинга.

Ниже — готовый короткий one-paragraph BRD:

Kanban Plugin for Agentic IDEs — это встроенный файловый kanban для управления агентской разработкой внутри IDE. Он автоматически находит источники задач, визуализирует workflow в нативном стиле хоста, синхронизируется с файлами проекта и позволяет запускать/контролировать выполнение задач через перемещение карточек, а не через хаотичное общение с агентами.

И готовый короткий PRD summary:

MVP должен работать в OpenCode Desktop, автоматически наследовать тему и шрифты, обнаруживать board/task-файлы, показывать колонки и карточки, поддерживать drag-and-drop, file watching, hotkeys и базовую интеграцию с оркестратором через изменение файлов и события смены статуса.

Составь на основании вот этого все необходимые документы и чтобы можно было запустить разработку.
```
