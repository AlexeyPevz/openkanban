# User Stories — OpenKanban for OpenCode

## Пользовательские сценарии

### Discovery и обзор

- Как пользователь, я хочу открыть kanban из OpenCode и сразу увидеть задачи текущего проекта без ручной разметки.
- Как пользователь, я хочу понимать, какой task source выбран автоматически, чтобы доверять board.
- Как пользователь, я хочу видеть список доступных проектов OpenCode внутри companion app, чтобы быстро смотреть, что происходит по другим проектам.

### Multi-project workflow

- Как пользователь, я хочу, чтобы companion app по умолчанию открывал активный проект OpenCode.
- Как пользователь, я хочу вручную переключиться в companion app на другой проект, не меняя active project в OpenCode.
- Как пользователь, я хочу, чтобы после ручного переключения проект оставался полноценно редактируемым, а не только для просмотра.

### Операционное управление

- Как пользователь, я хочу перетаскивать карточки между колонками и быть уверенным, что это меняет реальные файлы проекта.
- Как пользователь, я хочу менять статус карточки с клавиатуры, не только мышкой.
- Как пользователь, я хочу создавать новую карточку прямо из board workflow, чтобы не уходить в ручное редактирование файлов.
- Как пользователь, я хочу редактировать title, description и ключевую metadata карточки, чтобы поддерживать задачу в актуальном состоянии.
- Как пользователь, я хочу быстро переназначить агента или другой resource карточке, чтобы изменить strategy execution.

### Наблюдаемость

- Как пользователь, я хочу видеть, что делает orchestrator по карточке, не уходя в отдельные логи.
- Как пользователь, я хочу быстро понимать, где задача зависла и почему.
- Как пользователь, я хочу видеть, какие skills, resources и gates обязательны для задачи.

### Host-native опыт

- Как пользователь, я хочу открывать companion app командой или hotkey из OpenCode.
- Как пользователь, я хочу, чтобы theme и fonts companion app были близки к OpenCode host, а не выглядели как отдельный чужой инструмент.
- Как пользователь, я хочу, чтобы при наличии host flag companion app мог стартовать вместе с OpenCode.

### Recovery и предсказуемость

- Как пользователь, я хочу, чтобы board восстанавливался из файлов после перезапуска OpenCode или companion app.
- Как пользователь, я хочу, чтобы изменения файлов отражались в UI без ручного перезапуска.
- Как пользователь, я хочу, чтобы ошибка в одной карточке не ломала весь board.
- Как пользователь, я хочу предсказуемый degraded UX, если хост не даёт полный набор UI/command/event surfaces.

## Сценарии для разработчика harness

- Как разработчик harness, я хочу file-first source of truth, чтобы задачи были git-friendly и reviewable.
- Как разработчик harness, я хочу связать status transitions с orchestration bridge, чтобы kanban был управляющим слоем, а не витриной.
- Как разработчик harness, я хочу переиспользуемый core engine, чтобы потом добавить adapters для других agentic IDE.
- Как разработчик harness, я хочу локальный agent/resource registry + host-level registry + ad hoc entities, чтобы routing был гибким.
- Как разработчик harness, я хочу явный kanban execution contract, чтобы orchestrator и одиночный агент подчинялись одному lifecycle и не обходили board молча.
- Как разработчик harness, я хочу current-project binding + manual project switching, чтобы companion app был operational center, а не одноразовой проекцией active host context.

## Сценарии отказа и edge cases

- Как пользователь, я хочу видеть понятную причину, если preflight не даёт перевести задачу в `active`.
- Как пользователь, я хочу, чтобы board не терялся, если найдено несколько форматов задач.
- Как пользователь, я хочу, чтобы система показывала empty state, если task sources отсутствуют, а не молча ломалась.
- Как пользователь, я хочу, чтобы при недоступности host theme/font context app корректно переходил на fallback theme, не ломая workflow.
